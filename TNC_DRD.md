## 1. 프로젝트 개요 및 목표 (Context)

### 1.1 개발 목적
웹 브라우저 상의 정보를 팀 슬랙(Slack)으로 가장 빠르고 효율적으로 공유하는 **B2B 생산성 확장 프로그램**을 개발합니다. 복잡한 인증이나 앱 전환 없이, **'보고 있는 화면에서 즉시 요약 및 전송'**하는 경험을 제공하는 것이 핵심입니다.

### 1.2 비즈니스 전략 (Freemium)
* **MVP 단계:** 초기 진입 장벽을 낮추기 위해 **무료(Free)** 플랜 중심으로 개발하되, 향후 **유료(Pro)** 전환을 위한 기능적/UI적 확장성을 미리 확보해야 합니다.
* **우선순위:** '속도'와 '비용 효율성'을 최우선으로 하며, 초기에는 Webhook 기반의 연동을 통해 빠른 검증을 목표로 합니다.

## 2. 핵심 기능 요구사항 (Functional Requirements)

Tech Leader께서는 아래 기능을 구현하기 위한 최적의 아키텍처와 기술 스택을 선정해 주시기 바랍니다.

### 2.1 클라이언트 (Chrome Extension)
* **DOM 파싱 및 추출:**
    * 사용자가 보고 있는 웹페이지(로그인 세션 포함)의 본문 텍스트, 제목, URL을 정확하게 추출해야 합니다.
    * *제약:* 이미지 위주의 사이트나 SPA(Single Page Application)에서도 안정적인 추출이 필요합니다.
* **비용 방어 로직 (Cost Control):**
    * 확장 프로그램 실행(아이콘 클릭) 시 즉시 API를 호출하지 않고, **'대기 상태(Ready State)'** UI를 먼저 렌더링해야 합니다.
    * 사용자가 명시적으로 **[시작] 버튼을 눌렀을 때만** 백엔드 요약 API를 호출해야 합니다.
* **UI/UX:**
    * AI 분석 중 로딩 상태(Spinner/Skeleton) 처리 필수.
    * 요약 결과(텍스트, 태그)는 사용자가 수정 가능(Editable)해야 합니다.
    * **글자 수 제한(Counter)** UI가 실시간으로 동작해야 합니다 (Insight 150자, Comment 300자).

### 2.2 백엔드 및 AI (Server & AI)
* **LLM 처리:**
    * `OpenAI API` (권장: gpt-4o-mini)를 사용하여 텍스트 요약 및 인사이트 도출.
    * **프롬프트 분기 처리:** 사용자 설정(`mode`)에 따라 **'3줄 요약(줄글)'** 또는 **'키워드 추출(배열)'**로 응답 형식이 달라져야 합니다.
    * **API Key 보안:** 클라이언트 코드에 API Key가 노출되지 않도록 Proxy 서버 또는 Serverless Function을 경유해야 합니다.
* **전송 모듈 (Slack Sender):**
    * **Webhook 방식 (MVP):** 사용자가 입력한 Webhook URL로 `POST` 요청 전송.
    * **확장성 고려:** 추후 `OAuth` 기반의 슬랙 API 연동 및 `Discord` 등 타 메신저 확장이 용이하도록 **인터페이스(Interface) 기반 설계**를 권장합니다.

### 2.3 데이터 저장 및 관리
* **설정값 저장:** 사용자별 설정(Webhook URL, 요약 모드 등)은 브라우저 스토리지(`chrome.storage.sync` 등)를 활용하여 기기 간 동기화를 지원해야 합니다.
* **사용량 제한 (Quota):** (로컬 기준) 1일 무료 사용 횟수(예: 5회)를 카운팅하고 제한하는 로직이 필요합니다.

## 3. 기술적 제약 및 고려 사항 (Technical Constraints)

### 3.1 성능 및 속도 (Performance)
* **목표:** [요약 시작] 버튼 클릭 후 결과 노출까지 **평균 5초 이내**를 목표로 합니다.
* **스트리밍(Streaming):** 긴 텍스트 처리 시 체감 속도 향상을 위해 응답 스트리밍(Stream) 도입 여부를 검토해 주십시오.

### 3.2 보안 (Security)
* **개인정보:** 수집된 웹페이지 본문 데이터는 요약 생성 직후 메모리에서 파기되어야 하며, 별도 DB에 로그로 저장되지 않아야 합니다.
* **Webhook URL:** 사용자가 입력한 민감 정보(Webhook URL)는 안전하게 저장/전송되어야 합니다.

### 3.3 확장성 (Scalability)
* 현재는 Webhook 방식이지만, 추후 **OAuth 2.0 인증 서버** 구축이 예정되어 있습니다. 이를 고려하여 백엔드 구조를 잡아주십시오.
* 유료 플랜 도입 시 **기능 잠금(Feature Lock)** 로직을 클라이언트/서버 양단에서 처리할 수 있는 구조가 필요합니다.
